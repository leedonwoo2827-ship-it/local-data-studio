# 데이터 저장 전략 리포트 — "엑셀을 계속 쌓는다면 SQLite가 맞나?"

> 질문: 데이터를 누적할 수 있는가? 엑셀을 계속 넣을 텐데 SQLite가 좋은가?
> **결론: 이 시나리오에는 SQLite가 정답에 가깝다.** 본 스튜디오는 SQLite 누적 계층을 기본 탑재했다.

## 1. 요구사항 정리
- 팀별로 **주기적으로(매주/매월) 엑셀을 추가**하며 데이터를 **누적**한다.
- 로컬 PC에서 돈다. **서버를 깔고 운영할 인력이 없다.**
- 같은 기간을 실수로 다시 올려도 **중복으로 쌓이면 안 된다**(재현/정정 가능해야 함).
- 누적된 데이터로 대시보드/보고서를 그린다.

## 2. 선택지 비교

| 방식 | 누적 | 중복제거/수정 | 설치 | 동시성 | 적합도 |
|------|------|----------------|------|--------|--------|
| **CSV/엑셀 폴더에 계속 쌓기** | 파일만 늘어남 | 수작업, 사고 잦음 | 0 | 약함 | ❌ 관리 붕괴 |
| **SQLite (단일 .db 파일)** | 테이블에 append/upsert | SQL로 정확 | 0 (파이썬 내장) | 단일 사용자 OK | ✅ **권장** |
| DuckDB / Parquet | 분석 빠름 | 가능 | 패키지 | 분석특화 | 🔸 수억 행/분석 위주면 |
| PostgreSQL/MySQL 등 서버DB | 강력 | 강력 | 서버 운영 필요 | 다중 사용자 | 🔸 팀 공유·동시쓰기 본격화되면 |

### 왜 SQLite인가 (이 케이스)
- **설치 0**: 파이썬 표준 라이브러리(`sqlite3`). 추가 의존성·서버 없음.
- **단일 파일**: `db/lds.sqlite` 하나면 끝. 백업=파일 복사, 이동=파일 전송.
- **누적/정정에 강함**: append(추가) + **upsert(같은 기간 교체)** 로 재업로드가 안전(idempotent).
- **충분한 규모**: 수백만 행까지 무리 없음. 월별 KPI는 사실상 무한.
- **표준 SQL**: 나중에 PostgreSQL로 이전해도 쿼리 자산이 거의 그대로 간다.

### SQLite가 부족해지는 시점 (이전 신호)
- 여러 사람이 **동시에 쓰기**를 자주 한다 → 쓰기 잠금 경합. → PostgreSQL.
- 데이터가 **수천만~수억 행**, 복잡한 분석 쿼리가 느리다 → DuckDB/Parquet 또는 분석DB.
- **네트워크로 공유**되는 중앙 데이터셋이 필요 → 서버DB.

## 3. 본 스튜디오의 구현
- 코드: [core/store.py](../core/store.py)
- DB 파일: `db/lds.sqlite` (팀별 테이블 `data_<팀키>`, 팀키 = config 파일명)
- 적재 모드 3가지 (사이드바 **📥 데이터 적재** 패널):
  - **같은 기간 교체(upsert)** — 기본. 같은 `기간` 값 행을 지우고 새로 넣음 → 재업로드해도 중복 없음.
  - **무조건 추가(append)** — 일자 단위 원시 로그처럼 계속 덧붙일 때.
  - **전체 교체(replace)** — 스키마가 바뀌었거나 처음부터 다시.
- 적재 시 출처/시각 메타(`_source_file`, `_ingested_at`)를 자동 기록(조회 시엔 숨김).
- 사이드바 **데이터 소스**에서 `샘플 파일 / 누적 DB(SQLite) / 일회성 업로드` 전환.

### 사용 흐름
1. 사이드바 → **📥 데이터 적재** → 엑셀/CSV 업로드 → 모드 선택 → **DB에 적재**.
2. **데이터 소스 = 누적 DB(SQLite)** 선택 → 누적분 전체로 대시보드/보고서 생성.
3. 다음 달 엑셀이 나오면 다시 적재 → 자동 누적.

### 주의
- 업로드 파일의 **컬럼명이 config의 KPI `column`과 일치**해야 KPI가 계산된다(스키마 일관성).
  컬럼이 바뀌면 [docs/CONFIG_GUIDE.md](CONFIG_GUIDE.md)로 config를 갱신.
- 단일 PC/단일 사용자 가정. 팀 공유가 필요하면 위 "이전 신호" 참고.
